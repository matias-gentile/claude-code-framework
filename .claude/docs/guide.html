Open source framework

Agent-first engineering framework

A guide for people who've never used Claude Code before

What is this, really?
Imagine you're a construction project manager. You don't mix the concrete yourself — you have specialists: an architect who reads the plans, a safety inspector who checks the work, and tradespeople who do the actual building. You make the decisions; they handle the execution.
This framework does the same thing for software. Instead of you typing code, you tell Claude Code what needs building, and a team of specialized AI agents handles the execution. You stay at the level of decisions and direction.
Without the framework
You type code → Claude suggests → you type more → things break → you debug → repeat
With the framework
You describe what's needed → agents plan, write tests, implement, verify → you review decisions
The key mental shift
You are the engineering manager, not the coder. Claude's agents are your specialists. The framework is the set of standing rules they all follow — so you don't have to repeat yourself every session.
The framework is built from six layers. Click each one to see what it does.
CLAUDE.md — the constitution
Always loaded
This is the single most important file. Every time you open Claude Code, it reads this file first. Think of it as the standing orders for every agent — architectural rules, what agents to use and when, commit conventions, which skills exist. It's capped at 120 lines on purpose: research shows that beyond that, agents start missing rules.
  • Hard rules the AI must follow (no exceptions)
  • Escape hatches — when to stop and ask a human
  • Which agents and skills are available
  • References to longer docs in .claude/docs/
Skills — on-demand knowledge
8 skills
Skills are like reference manuals that Claude reads only when needed. If you're creating a REST API endpoint, Claude automatically loads the api-conventions skill. If something is broken, it loads the runbook. This keeps the token cost low — only the relevant knowledge loads.
  • setup + tutorial — onboarding (run once)
  • verification — end-to-end feature confirmation
  • tdd-practices — test-first enforcement
  • runbook — structured debugging protocol
  • session-notes — structured record-keeping
  • adr-recorder — capturing decisions
  • api-conventions — REST design rules
Agents — your specialist team
4 agents
Instead of one general-purpose AI doing everything, you have four specialists. Each has a specific job, specific tools, and runs in an isolated environment so a mistake in one can't affect the others.
  • planner — reads the codebase, never writes code
  • tdd-writer — writes failing tests only
  • code-reviewer — reviews changes, never writes code
  • quality-gate — checks everything is really done
Hooks — automatic quality gates
3 hooks
Shell scripts that fire automatically at key moments — you don't have to remember to run them. They're the safety net that catches problems before they become commits.
  • After every file write → runs your test suite automatically
  • Before every git commit → blocks it if tests haven't passed
  • When Claude finishes a task → reminds you to capture lessons
MCP — external connections
Optional
MCP (Model Context Protocol) lets Claude connect to external tools — GitHub, databases, live documentation. Your tokens are stored locally in .mcp.json which is gitignored, so they never get committed.
  • GitHub — read PRs, issues, repo context
  • Context7 — live docs (no more hallucinated APIs)
  • Database — read-only query access for debugging
ADRs — decision history
Permanent record
Architecture Decision Records. When you make a significant technical decision, you record it as a short markdown file. Next time anyone works on that area, the relevant ADR is read first — so decisions aren't accidentally reversed.
  • Numbered files: adr/0001-framework-setup.md
  • The adr-recorder skill creates them for you
  • Keeps CLAUDE.md short — details live in ADRs
Four specialists, each with a specific job. You dispatch them by typing a command in Claude Code.
planner
Phase 1 — reads before anyone writes
Scouts the codebase, maps what will be affected, produces a numbered plan for you to review. Has no write access — it physically cannot change files.
When: before any task touching more than one file
Type: Use the planner agent to explore [task]
tdd-writer
Phase 3 start — defines correctness first
Writes failing tests before implementation exists. Cannot write to src/ — tests only. Forces you to define what "correct" means before building.
When: before writing any feature code
Type: Use the tdd-writer agent to write tests for [behaviour]
code-reviewer
Phase 4 — gates every merge
Reviews code for architecture, security, and correctness. Has no write access. Returns a verdict: approved, approved with notes, or blocked.
When: before every merge or PR
Type: Use the code-reviewer agent to review this change
quality-gate
Second opinion — are you really done?
Checks six things before you close a task: tests pass, verification confirmed, plan complete, no stray TODOs, notes current, commit scoped correctly.
When: before closing any task
Type: Use the quality-gate agent to evaluate this task
Skills are knowledge modules that load only when relevant — keeping Claude's memory clean. You get 8 out of the box.
setup
Scans your codebase on day one and auto-fills all placeholder files. Run once.
tutorial
Interactive 8-step walkthrough inside Claude Code using your actual project. About 20 minutes.
verification
End-to-end flows for your most important features. Proves the feature works from the user's perspective.
tdd-practices
Test-first rules. Defines the sequence: failing test → implementation → green.
runbook
Structured debugging protocol. Systematic investigation instead of guess-and-check.
session-notes
Structured notes written during work. Summaries come from the notes, not from Claude's memory.
adr-recorder
Captures architectural decisions into numbered files in adr/.
api-conventions
REST design rules: kebab-case URLs, camelCase JSON, pagination, status codes.
Every task that touches more than one file follows this 4-phase flow. It's enforced in CLAUDE.md so Claude follows it automatically.
1
Explore (read-only)
Dispatch the planner agent. It reads the codebase, maps everything that will be affected, identifies risks, and produces a numbered plan. You review and approve before anything is written.
planner agent
2
Plan (approved by you)
You review the planner's numbered task list. Approve, modify, or ask questions. Nothing is implemented until you give the go-ahead. This is your most important moment of control.
human decision
3
Implement (tests first)
Dispatch tdd-writer to write failing tests that define what "correct" looks like. Then implement the code to make those tests pass. Hooks run tests automatically after every file write.
tdd-writer → implement → hooks
4
Verify (done means confirmed)
Run the verification skill to confirm the feature works end-to-end. Dispatch the code-reviewer before merging. Dispatch the quality-gate to confirm everything is complete.
verification + reviewer + quality-gate
The compounding loop is the most important habit. Every session makes the next one faster. The session-end hook fires these questions automatically.
Task happens
You build something, fix a bug, or make a decision
Ask 3 questions
Did Claude get something wrong? Was there a decision? Did a workflow take many turns?
Codify the lesson
Add a rule to CLAUDE.md, create a skill, or record an ADR
Next session is faster
The framework remembers so you don't repeat yourself
The three questions (fired automatically at session end)
1. Did Claude get something wrong repeatedly? → Add a rule to CLAUDE.md
2. Was there an architectural decision? → Use the adr-recorder skill
3. Did a workflow take many turns? → Create a new skill
Teams that do this consistently go from 1.5× productivity on day one to 5–10× by month two. Teams that skip it stay at 1.5× forever.
Three kinds of safety nets prevent the most common ways AI-assisted development goes wrong.
1. Escape hatches — stopping stuck agents
Same tool output 3 times in a row
You are in a loop — STOP, change approach or ask human
Same fix attempted 3 times, no progress
STOP, summarize what you tried, ask the human
10 debugging calls with no hypothesis
Invoke the runbook skill immediately
15 tool calls on a single sub-task
Check in with the human before continuing
2. Hooks — automatic enforcement
After every file write → runs your test suite, marks pass/fail
Before every git commit → blocks if tests haven't passed
When Claude finishes → fires the compounding loop reminder
3. Security defaults
.mcp.json is gitignored — API tokens can never be committed
Deny list in settings.json — blocks rm -rf, sudo, writing to .env or *.key
Agent isolation — each agent runs in its own git worktree
Five steps from zero to a working framework. Each step handles its own prerequisites.
1
Copy framework into your project root
Clone to a temp location, then copy the contents (including hidden files) directly into your project root. The trailing /. is important — it copies hidden folders like .claude/.
git clone https://github.com/matias-gentile/claude-code-framework.git /tmp/claude-framework rm -rf /tmp/claude-framework/.github rm -f /tmp/claude-framework/README.md cd your-project cp -r /tmp/claude-framework/. . rm -rf /tmp/claude-framework
Already have .claude/ files? Back them up first: cp -r .claude/ .claude-backup/
Want the GitHub Actions PR review? Bootstrap will ask you about it and walk you through setup.
2
Follow the installer prompts
The installer backs up existing configs, configures your test command, appends to your .gitignore, and optionally adds the GitHub Actions PR review workflow.
# The installer handles everything — just answer the prompts
3
Open Claude Code
Claude reads CLAUDE.md automatically and loads all agents and skills.
claude
4
Run the setup skill
Claude scans your codebase and fills in every placeholder automatically. Creates 3–5 verification flows.
Run the setup skill
5
Run the tutorial skill
Interactive 20-minute walkthrough using your real project. Teaches by doing, not by explaining.
Run the tutorial skill
That's it.
The framework is now running. Every session after this, just open Claude Code in your project and it picks up where you left off.
