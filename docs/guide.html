The problem

AI assistants are powerful but undisciplined.

Most teams use Claude Code the same way they used Stack Overflow — ask a question, get an answer, move on. The AI has no memory of your architecture, no standing rules to follow, and no way to get better at your specific project over time. Every session starts from zero.

Without the framework
  • You repeat context every session
  • AI ignores your architecture conventions
  • No tests written → bugs ship
  • Decisions made in chat, forgotten tomorrow
  • Agents loop on problems, waste hours
  • Credentials accidentally committed
With the framework
  • Rules persist across every session
  • Architecture enforced automatically
  • Tests written before any implementation
  • Decisions recorded as permanent ADRs
  • Escape hatches break infinite loops
  • Credentials never touch version control
The shift

Six layers

A structured system, not a chat interface.

The framework adds six layers on top of Claude Code. Each layer has one job. Click any to expand.

1
CLAUDE.md — the rules file
Always loaded · ≤120 lines
Read at the start of every session, automatically. This is where your standing orders live — architectural rules, which agents to use when, commit conventions, active skills. Capped at 120 lines because research shows agents miss rules beyond that limit.

Automatically enriched by the setup command, which scans your codebase and extracts project-specific rules across 5 categories: module boundaries, wrapper conventions, naming surprises, data shape invariants, and error handling patterns.
2
Skills — on-demand knowledge
8 skills · loads when relevant
Knowledge modules that load only when the topic is relevant — saving memory for actual work.
  • setup — scans codebase, fills placeholders, enriches CLAUDE.md
  • verification — confirms features work from the user's perspective
  • tdd-practices — test-first enforcement
  • runbook — structured debugging protocol
  • session-notes — writes notes so summaries come from records, not memory
  • adr-recorder — captures architectural decisions permanently
  • api-conventions — REST design rules
3
Agents — specialist workers
4 agents · isolated execution
Four specialists, each with a specific job and restricted tools. They run in isolated environments — a mistake in one cannot affect the others.
  • planner — reads the codebase, produces a plan, never writes code
  • tdd-writer — writes failing tests before implementation starts
  • code-reviewer — reviews for architecture and security, never edits files
  • quality-gate — runs 6 checks before any task closes
4
Hooks — automatic lifecycle automation
6 hooks · deterministic
Shell scripts that fire at lifecycle events — they don't rely on the AI deciding to do the right thing. The framework covers six of Claude Code's lifecycle events:
  • SessionStart — loads prior context: current branch, last session's handoff notes, most recent decision, and your test command. This is what carries knowledge forward between sessions.
  • UserPromptSubmit — blocks prompts containing secrets (API keys, tokens) and injects the relevant past decision when your prompt touches a decided area.
  • PostToolUse — after every file write, runs the matching test automatically.
  • PreToolUse — before every git commit, blocks it if tests haven't passed.
  • PreCompact — before context is compacted, prompts a save of session notes so nothing is lost.
  • Stop — when Claude finishes, prompts the session-review command to capture lessons.
5
MCP — external connections
Optional · tokens gitignored
Connects Claude directly to GitHub, live library documentation, and databases. Credentials live locally in .mcp.json — gitignored, never committed. Pre-configured: GitHub (PRs and issues), Context7 (live docs that eliminate API hallucinations), read-only database access.
6
ADRs — decision history
Permanent · auto-referenced
Architecture Decision Records. Every significant technical decision gets recorded in .claude/adr/. Agents read the relevant ADR before touching that area — so decisions are never accidentally reversed. Created automatically by the session-review command when it detects a decision was made.
Agents

Four specialists. Each with one job.

Instead of a general-purpose AI doing everything, you dispatch specialists. Each has restricted tools — the planner physically cannot edit files. The reviewer physically cannot write code.

planner
Phase 1 — before any code
Reads the codebase, maps everything affected, checks ADRs for constraints, produces a numbered plan for your approval.
Cannot write or edit any file.
Command: plan [task]
tdd-writer
Phase 3 start — tests define done
Writes failing tests that define what "correct" means before a line of implementation exists. Forces clarity on acceptance criteria first.
Cannot write to source directories.
Say: Use the tdd-writer agent to write tests for [behaviour]
code-reviewer
Phase 4 — before every merge
Reviews changes for architectural alignment, security, and test coverage. Returns Approved, Approved with notes, or Blocked.
Cannot write or edit any file.
Command: review
quality-gate
Second opinion — really done?
6-point checklist before a task closes: tests pass, verification confirmed, plan complete, no new TODOs, notes current, commit clean.
Cannot write or edit any file.
Say: Use the quality-gate agent to evaluate this task
Commands

Six slash commands. Available everywhere.

All framework actions are slash commands — visible in the / menu in Claude Code. No documentation needed to discover them.

The prefix depends on how you installed: /agent-first-framework:setup in plugin mode, /project:setup in copy mode. The command names below are shown without a prefix.

setup
Scan the codebase, fill placeholders, enrich CLAUDE.md with project-specific rules, create verification flows. Run once after installing.
tutorial
20-minute interactive walkthrough using your real project. Teaches by doing — dispatches agents on real tasks.
plan
Dispatch the planner agent — explore the codebase and produce an implementation plan before any code is written.
review
Dispatch the code-reviewer agent on current changes before any merge or pull request.
adr
Record an architectural decision into a permanent file in .claude/adr/.
session-review
Draft lessons from this session — rules, ADRs, skills — for your approval. Run before /clear. This is what makes the framework compound.
update
Pull the latest framework improvements into this project. Updates components you haven't touched, preserves your customizations, and never alters your CLAUDE.md.
The 4-phase flow

Every task. Same sequence.

Every task touching more than one file follows this sequence — enforced in CLAUDE.md so Claude follows it automatically. Most developers skip straight to phase 3. That's where most rework comes from.

1
Explore — read before writing
The planner agent reads the entire affected surface. It maps which files change, checks relevant ADRs for constraints, identifies risks, and produces a numbered plan. You approve before anything is written. Takes 2 minutes. Saves 30 minutes of rework.
plan → planner agent
2
Plan — your approval gates everything
You review the numbered task list. Approve, modify, or ask questions. Nothing is implemented until you say so. This is the most important moment of human control in the process.
Human decision point
3
Implement — tests define done
The tdd-writer writes failing tests that define what "correct" looks like before implementation starts. Then code is written to make those tests pass. Hooks run the test suite automatically after every file write.
tdd-writer → implement → hooks auto-test
4
Verify — done means confirmed, not just passing
The verification skill confirms the feature works from the user's perspective — not just that unit tests pass. The code-reviewer checks architecture and security. The quality-gate runs a 6-point checklist before the task closes.
review + quality-gate + verification

What gets captured and how
Rules → CLAUDE.md
If Claude got something wrong and had to be corrected, session-review drafts the exact rule text with rationale and budget impact. You approve it. Next session, Claude follows it automatically.

Example: "Never import axios directly — use httpClient from src/lib/http.ts"
Decisions → Architecture Decision Records
If an architectural decision was made, a full ADR is drafted — context, decision, alternatives considered, consequences. Agents read it before touching that area in future sessions.

Example: ADR-0003 "Cursor-based pagination — rejected offset for performance"
Workflows → New skills
If a workflow took many turns to get right, a new skill is drafted — name, description, content preview — for your approval. That workflow becomes a one-turn interaction next time.

Example: A deployment verification checklist that took 8 turns to establish
Recurring roles → Custom agents
If a session surfaced a distinct repeating role — database migrations, API client generation, localization — session-review proposes a new specialist agent with its own restricted tools and contracts, for your approval. The framework grows its own workforce based on your actual project.

Example: A db-migrator agent proposed after three sessions of migration work

Safety nets

Three layers of protection.

The most common ways AI-assisted development goes wrong: agents that loop forever, broken code that ships, and credentials that get committed. The framework has specific mechanisms for each.

Escape hatches — stopping stuck agents
Same tool output 3 times in a row
Agent is in a loop — forced to stop, change approach, or escalate to human
Same fix attempted 3 times, no progress
Agent must summarize what it tried and ask the human for direction
10 debugging calls with no clear hypothesis
Runbook skill invoked — structured investigation replaces guessing
15 tool calls on a single sub-task
Human check-in required before continuing
Automatic quality enforcement
SessionStart — context carried forward
At the start of every session, the branch, last session's handoff notes, the most recent architectural decision, and your test command are loaded automatically. The framework remembers across sessions.
UserPromptSubmit — secret blocking + decision injection
Prompts containing API keys, tokens, or private keys are blocked before Claude ever sees them. When your prompt touches an area with a recorded decision, that decision is surfaced automatically.
After every file write
The matching test runs automatically. Claude sees failures and fixes them before moving on — not after the whole feature is built.
Before every git commit
The commit is blocked if tests haven't passed since the last edit. Broken code cannot be committed regardless of how the commit is triggered.
Before compaction
When context is about to be summarized away, a save of session notes is prompted first — so decisions and progress survive, generated from facts rather than fading memory.
Security defaults
.mcp.json is gitignored
API tokens and database credentials live in a local file explicitly excluded from version control. They cannot be accidentally committed.
Deny list in settings.json
Destructive commands (rm -rf, sudo), credential files (.env, *.key, *.pem), and infrastructure directories are blocked regardless of what the AI is asked to do.
Agent isolation
Each agent runs in its own isolated git worktree. A mistake or bad output from one agent cannot corrupt another agent's working state.
Getting started

Two ways to install. Pick one.

The framework installs either as a Claude Code plugin (recommended — one command, easy to update) or copied directly into a single project. Both deliver identical functionality.

Option A — As a plugin (recommended)
1
Add the marketplace and install
Requires Claude Code v2.1 or later. The plugin's components live in Claude Code's plugin cache — nothing is copied into your repo yet.
/plugin marketplace add matias-gentile/claude-code-framework /plugin install agent-first-framework@claude-code-framework
2
Run setup — required
In plugin mode this step is mandatory. It creates CLAUDE.md (the framework's rules and 4-phase flow) from a template, then enriches it with your project's conventions. Without it you have the tools but not the rules. Commands are namespaced with the plugin name.
claude /agent-first-framework:setup
Option B — Copied into one project
1
Clone and run the installer
Copies 3 files to your project root and everything else into .claude/. Never overwrites your existing files; backs up any existing .claude/ config; appends to your .gitignore; detects your test runner.
git clone https://github.com/matias-gentile/claude-code-framework.git /tmp/claude-framework cd your-project bash /tmp/claude-framework/install.sh rm -rf /tmp/claude-framework
2
Run setup
CLAUDE.md already exists from the installer, so setup enriches it with project-specific rules and configures everything else. Commands use the /project: prefix.
claude /project:setup
Then, in either mode
3
Tutorial — optional
A 20-minute interactive walkthrough using your real project. Skip it if you already know the framework. It teaches; it doesn't configure anything setup didn't already handle.
/agent-first-framework:tutorial (plugin) /project:tutorial (copy)
4
Session-review at the end of each session
Before you /clear context, run the session-review command. It reads what happened, drafts rules, ADRs, or new agents worth keeping, and asks for approval. The Stop hook reminds you automatically. This is what makes the framework compound over time.
/agent-first-framework:session-review (plugin) /project:session-review (copy)
Which should I choose?
Use the plugin if you want the framework across many projects and easy updates — it's the modern Claude Code distribution method. Use copy if you want the framework's files committed directly into one project's repo so the whole team gets them via version control.
Updating later

When the framework ships improvements, pull them in without losing your work. In copy mode, run the update command or the updater script; in plugin mode, reinstall the plugin. The updater refreshes components you haven't modified, leaves anything you customized untouched (saving the new version beside it as .new to compare), and never alters your CLAUDE.md, settings, or notes.

git clone https://github.com/matias-gentile/claude-code-framework.git /tmp/cf bash /tmp/cf/update.sh rm -rf /tmp/cf
Already have your own Claude Code setup?

The framework is built to coexist. If you already have your own agents, skills, commands, hooks, or settings, none of it is destroyed. Your files with unique names live alongside the framework's. If a component collides by name (say you have your own planner), your version is kept and the framework's is saved as .framework for you to compare. Your settings.json is merged — your permissions and hooks are preserved, the framework's are added only where missing. And your existing CLAUDE.md is reconciled with your consent: setup appends the framework sections it's missing without touching your content.
